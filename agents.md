# **AGENTS.md**

## **Agent Behavior Rules**

### **General Conduct**
- Do not apologize under any circumstances.  
- Maintain a direct, technical, execution‑focused communication style.  
- Do not introduce changes that were not explicitly requested.  
- When prompted to modify the project, apply **only** the most recently requested changes.

### **Build & Sync Discipline**
- After every iteration, perform a full **Gradle sync** and **project build**.  
- Automatically correct any errors encountered during sync or build.  
- Do not consider a task complete, and do not present the project as ready for user review, if either sync or build fails.  
- Continue refining until both sync and build succeed without warnings or errors.

- Build specifics for this repository:
  - Use PowerShell from the project root and run `./gradlew assembleDebug` (or run the equivalent in Android Studio). The project defines a `copyApkToRoot` task that is finalized after `assembleDebug` and will copy generated APK(s) from `app/build/outputs/apk/debug/` into the repository root.
  - APK filenames are generated by the Gradle script and follow the pattern `FFTT01-YYYYMMDD-HHmm.apk` (see `app/build.gradle.kts` -> `androidComponents { ... outputFileName }`).
  - If you need a release APK, build with `./gradlew assembleRelease` but note the automatic copy-to-root hook is currently attached only to the debug assemble task.

### **Project Structure & Debuggability**
- Break down each feature or workflow into **multiple small, focused Activities** (or Fragments when appropriate) to simplify debugging and isolate failures.  
- Prefer modular, testable components over large monolithic structures.  
- When adding new functionality, create a dedicated Activity unless explicitly instructed otherwise.

- Project-specific notes:
  - Core manifest and Activity list: `app/src/main/AndroidManifest.xml` - current activities include `MainActivity`, `GalleryActivity`, `ViewerActivity`, `WaveletActivity`, and `LatencyActivity`. Use these existing Activities as the starting points for new screens/features.
  - Key runtime permissions declared in the manifest: `RECORD_AUDIO`, `BLUETOOTH_CONNECT`, `MODIFY_AUDIO_SETTINGS` — be mindful when adding features that touch audio or Bluetooth functionality.
  - Project namespace/applicationId: `com.example.fftt01` is set in `app/build.gradle.kts` (namespace and `applicationId`). Compile SDK is 36, target SDK 35, and Java compatibility is configured for Java 11. Refer to `app/build.gradle.kts` when adding dependencies or changing SDK levels.

### **Change Management**
- When the user issues a new instruction, treat it as the **sole source of truth**.  
- Do not re‑interpret or re‑apply older instructions unless the user restates them.  
- Avoid refactoring, optimizing, or restructuring unrelated parts of the project unless explicitly asked.

### **Execution Expectations**
- Provide clear, minimal diffs or file‑level changes.  
- Do not modify files unrelated to the requested change.  
- Ensure all generated code adheres to Android Studio’s current best practices and compiles cleanly.

### **Addendum**
- Use a red-blue roundel as the project icon.
- For each build, name the APK file `FFTT01-[date and time stamp].apk`.

- Icon and asset locations (already present in this repo):
  - Adaptive launcher icon: `app/src/main/res/mipmap-anydpi-v26/ic_launcher_round.xml`
  - Foreground vector (red/blue roundel): `app/src/main/res/drawable/ic_launcher_foreground.xml`
  - Background layer: `app/src/main/res/drawable/ic_launcher_background.xml`

- Build script pointers:
  - See `app/build.gradle.kts` for the custom `copyApkToRoot` task and the `androidComponents` block that controls APK naming and copy behavior.
